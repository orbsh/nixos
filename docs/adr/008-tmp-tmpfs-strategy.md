# ADR-008: /tmp tmpfs 挂载策略与风险评估

**日期**: 2026-06-05
**状态**: 已采纳

### 问题

NixOS 在执行 `nixos-rebuild` 构建系统时，会在 `/tmp` 产生数 GB 的高频临时写入。将 `/tmp` 挂载到内存（tmpfs）可避免 SSD 磨损，但会带来突发性内存争抢风险。需要针对不同主机类型制定合理的策略。

### 核心工作机制

tmpfs 是"用多少，吃多少"的动态分配：

- **不提前占用**：设置的大小上限（如 `"50%"` 或 `"16G"`）只是安全红线。系统刚启动或 `/tmp` 空置时，几乎不消耗任何物理内存。
- **动态伸缩**：临时文件写入时内存按需增长；编译结束、文件删除后，内存立刻释放归还给系统。

### 风险分析

尽管平时不占内存，但在系统更新（`nixos-rebuild switch`）或编译大型软件时，会带来突发性的内存争抢危险：

| 风险 | 说明 |
|------|------|
| OOM 崩溃 | 服务器业务已占用大部分内存时，`/tmp` 瞬间暴涨会引发内存耗尽，内核可能强制杀死核心业务进程 |
| 物理硬盘无寿命焦虑 | 企业级 SSD 的 TBW 极高，相比之下 99.99% 的业务稳定性远比延长硬盘寿命重要 |

### 决策

按主机类型分层决策：

#### Workstations — 开启 tmpfs（已实施）

工作站内存通常充沛，且频繁进行本地 Nix 构建。在 `modules/presets/workstation.nix` 中配置：

```nix
boot.tmp.useTmpfs = true;
boot.tmp.tmpfsSize = "50%";
```

#### Server — 保持默认（不开启）

服务器内存可能被业务大量占用，稳定性优先于 SSD 寿命。保持 `useTmpfs = false`（默认值），配合 `noatime` + `fstrim` 即可满足日常寿命优化。

#### 服务器折中方案（特殊情况）

如果服务器内存极其充沛（64GB+）且有大量结余，可以安全开启，但建议：

```nix
boot.tmp.useTmpfs = true;
boot.tmp.tmpfsSize = "16G";  # 固定大小，不按百分比，守住业务内存底线
```

并配置充足的 Swap 作为防爆保底。

### 后果

| 主机类型 | tmpfs | 理由 |
|----------|-------|------|
| workstations (orbit/alice/bob) | ✅ `50%` | 内存充沛，频繁构建，SSD 寿命敏感 |
| server | ❌ 不开启 | 稳定性优先，企业级 SSD 无寿命焦虑 |
| portable | ❌ 不开启 | 移动设备内存有限，稳定性优先 |
| qemu | ❌ 不开启 | 虚拟机内存受限 |
| k8s 集群 | ❌ 不开启 | 节点内存需预留给业务 Pod |

### 理由

1. **场景适配** — 工作站构建频繁且内存充裕，tmpfs 收益最大
2. **稳定性优先** — 服务器 99.99% 可用性远高于 SSD 寿命优化
3. **固定大小兜底** — 服务器若开启，用固定值而非百分比，避免挤占业务内存
4. **分层覆盖** — 仅在 workstation preset 中开启，其他主机不受影响
